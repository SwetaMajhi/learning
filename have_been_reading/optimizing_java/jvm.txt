Interpreting and Classloading in Java

1. covers how the JVM runs your code and how it finds and loads the classes needed to run it.

1. The JVM is a stack-based machine — instead of using hardware registers (like a real CPU), it uses a stack to do calculations.
2. What happens when you run java HelloWorld:
    - The OS starts the JVM process
    - JVM sets up the stack machine environment
    - It looks for the main() method in HelloWorld.class
3. Before executing, it must load the class = classloading. When JVM starts, it uses a chain of 3 classloaders.
    - There are 3 classloaders in a parent-child chain: Bootstrap -> Extension -> Application
    - When a class is needed and not found by classloaders -> they delegate to parent. 
    - If eventually Bootstrap classloader is unable to find it -> ClassNotFoundException is thrown
    - A class is generally loaded only once. But the same class can be loaded twice by different classloaders. Hence, class is uniquely identified by: classloader + fully qualified class name (package + class name)

4. Executing Bytecode
    - The job of javac is to convert Java code into .class files that contain bytecode.
    - Very few optimizations are done during compilation by javac
    - Bytecode is an intermediate representation that is not tied to a specific machine architecture = Decoupled from the machine architecture = already developed (or compiled) software can run on any platform supported by the JVM
    - JVM can execute any JVM language that can produce a valid class file = the Scala compiler scalac can generate bytecode for execution on the JVM.
    - Any class that is loaded by the JVM will be verified to conform to the expected format before being allowed to run.
    - Every class file starts with the magic number 0xCAFEBABE, and a part of it is checked (by the classloader) to ensure that the target JVM is not of a lower version than the one used to compile the class file. If these are not compatible an UnsupportedClassVersionError will be thrown at runtime, indicating the runtime is a lower version than the compiled class file.

5. Hostpot
    - HotSpot is the JVM engine that dramatically improved Java's performance.
    - two different philosophies in how programming languages are built:
        - Close to the metal: full control, maximum raw speed, zero overhead = Code must be compiled to platform-specific machine code at build time — called Ahead-of-Time (AOT) compilation.
        - Developer productivity: Easier to write, portable, runtime handles optimizations = Java approach = HotSpot observes which parts of your code run frequently (called "hot spots") and applies intelligent optimizations there automatically.

6. JIT compilation
    - compiler turns .java into .class files containing bytecode. Then JVM interprets this bytecode. This gives portability (run anywhere) but sacrifices raw speed because code isn't running directly on the CPU.
    Like how  C code ultimately turns into machine code that the computer's CPU can directly execute, but in case of java, the CPU is actually running the JVM interpreter program, which in turn reads and processes your Java bytecode. Like watching eng movie vs watching gujrati movie with subtitles. 
    - The goal: Get the best of both worlds — portability AND near-native CPU performance. JIT compilation is how HotSpot achieves this.
    - Hotspot compiles code while the program is running, not before.
    - HotSpot watches your running program and converts frequently used bytecode to native machine code on the fly.
    - Units that get JIT compiled = Methods & Loops
    - How JIT Works 
        - Program Starts
        - Runs in interpreter mode (slow but observing)
        - HotSpot monitors which methods/loops run most often 
        - Collects "trace information" (profiling data) 
        - When a method crosses a threshold (runs enough times)
        - HotSpot compiles that method to native machine code
        - Next time, runs directly on CPU = faster
    Hence HotSpot made more informed optimizations = can even detect the precise CPU type it is running on at VM startup, and can use this information to enable optimizations designed for specific processor features if available. 
    - After being translated from Java source to bytecode and now going through another step of (JIT) compilation, the code actually being executed has changed very significantly from the source code as written. JIT-compiled code executing on the JVM may well look nothing like the original Java source code.
    - ** Analyzing the performance of small sections of Java code (microbenchmarks) is usually actually harder than analyzing entire applications, and is a very specialized task that the majority of developers should not undertake.

7. JVM Memory Management
    - Poor manual memory management was a major cause of bugs and errors in languages like C and C++, wasting developer time. Hence, Java looked to help resolve the problem by introducing automatically managed heap memory using a process known as garbage collection (GC).
    - GC triggers when the JVM needs more memory = nondeterministic = can't predict exactly when it will run.
    - It recovers and reuses memory that is no longer needed. 
    - GC comes with a cost = when it runs, it causes a "Stop-the-World" (STW) pause = entire application freezes/pauses while GC is doing its work. These pause times are designed to be very small but they still exist

8. Threading and the Java Memory Model
    - Languages like C and C++ had multithreading but it was complex, platform-specific and not standardized, Java made it simpler and portable = same multithreading code runs on any platform.
    - Each Java application thread maps to a dedicated OS thread (1:1 mapping). Green threads (shared pool approach) was tried but rejected due to poor performance and added complexity.
    - The Java Memory Model (JMM) is a formal model of memory that explains how different threads of execution see the changing values held in objects. That is, if threads A and B both have references to object obj, and thread A alters it, what happens to the value observed in thread B?
        - This is complex because the OS scheduler can kick a thread off the CPU at any time — another thread may see an object in a damaged/invalid state.
        - Java provides mutual exclusion locks as the core defense against concurrent object damage. But locks are complex to use correctly in real applications.
