Minimize mutability
- An immutable class is simply a class whose instances cannot be modified. All of the information contained in each instance is fixed for the lifetime of the object, so no changes can ever be observed
- String, BigInteger, BigDecimal.

-To make a class immutable
    - Don’t provide methods that modify the object’s state (known as mutators).
    - Ensure that the class can’t be extended, by making the class final,
    - Make all fields final.
    - Make all fields private: While it is technically permissible for immutable classes to have public final fields containing primitive values or references to immutable objects, it is not recommended because it precludes changing the internal representation in a later release
    - Ensure exclusive access to any mutable components. Make defensive copies in constructors, accessors, and readObject methods.

- Example:
// Immutable complex number class 
public final class Complex { 
    private final double re;
    private final double im;
    public Complex(double re, double im) { 
        this.re = re; this.im = im; 
    }

    public Complex plus(Complex c) { 
        return new Complex(re + c.re, im + c.im); 
    }
}
    - Arithmetic operations create and return a new Complex instance rather than modifying this instance
    - This pattern is known as the functional approach: methods return the result of applying a function to their operand, without modifying it. 
    - Procedural or imperative approach: methods apply a procedure to their operand, causing its state to change.
    - The method names are prepositions (such as plus) rather than verbs (such as add). This emphasizes the fact that methods don’t change the values of the objects.
    - The BigInteger and BigDecimal classes did not obey this naming convention, and it led to many usage errors. 
    The most frequent mistake with BigDecimal is forgetting that instances are immutable, leading developers to call methods like add() without capturing the returned value.
        BigDecimal balance = new BigDecimal("100.00");
        BigDecimal deposit = new BigDecimal("50.00");

        // WRONG - balance is still 100.00!
        balance.add(deposit);  // Returns new object, but we ignore it

        System.out.println(balance);  // Still prints 100.00 - BUG!

- Thread safe
    -  Immutable Objects = Thread-Safe by Default = No synchronization needed
    - Cannot be corrupted - Multiple threads can't create inconsistent state = Complete isolation
    - Share freely - Pass immutable objects around without defensive copying
        public class Person {
            private String firstName;
            private String lastName;
            
            public void setName(String first, String last) {
                this.firstName = first;  // Step 1
                this.lastName = last;     // Step 2
            }
            
            public String getFullName() {
                return firstName + " " + lastName;
            }
        }
        
        
        //Thread corruption scenario in main code:
        Person person = new Person();
        person.setName("John", "Doe");

        // Thread 1: Changes name to Alice Smith
        new Thread(() -> person.setName("Alice", "Smith")).start();

        // Thread 2: Changes name to Bob Jones
        new Thread(() -> person.setName("Bob", "Jones")).start();

        // Thread 3: Reads name
        new Thread(() -> System.out.println(person.getFullName())).start();

        // Possible outputs (all corrupted except first):
        // "John Doe"      ← Original (if Thread 3 runs first)
        // "Alice Smith"   ← Thread 1 completed
        // "Bob Jones"     ← Thread 2 completed
        // "Alice Jones"   ← CORRUPTED! (Thread 1 set firstName, Thread 2 set lastName)
        // "Bob Smith"     ← CORRUPTED! (Thread 2 set firstName, Thread 1 set lastName)
        
        public final class ImmutablePerson {
            private final String firstName;
            private final String lastName;
            
            public ImmutablePerson(String firstName, String lastName) {
                this.firstName = firstName;
                this.lastName = lastName;
            }
            
            public String getFullName() {
                return firstName + " " + lastName;
            }
            
            // Returns new object instead of modifying
            public ImmutablePerson withName(String first, String last) {
                return new ImmutablePerson(first, last);
            }
        }

        //main code:
        ImmutablePerson person = new ImmutablePerson("John", "Doe");

        // Thread 1: Creates NEW person "Alice Smith"
        ImmutablePerson p1 = person.withName("Alice", "Smith");

        // Thread 2: Creates NEW person "Bob Jones"
        ImmutablePerson p2 = person.withName("Bob", "Jones");

        // Thread 3: Reads ORIGINAL person
        System.out.println(person.getFullName());  // Always "John Doe"

        // Each thread has its own object - no interference!
        // Original person is untouched
        // No corrupted state possible      
    
    - Provide public static final constants for commonly used values
        // Java's String class does this
        public final class String {
            // Empty string is commonly used - provide a constant
            public static final String EMPTY = "";  // Single instance shared by all
        }

        // Usage
        String s1 = "";           // Creates new object
        String s2 = String.EMPTY; // Reuses existing object (more efficient)

    - Share freely - Pass immutable objects around without defensive copying
        public final class ImmutableConfig {
            private final List<String> allowedHosts;
            
            public ImmutableConfig(List<String> hosts) {
                this.allowedHosts = List.copyOf(hosts);  // Immutable copy
            }
            
            public List<String> getAllowedHosts() {
                return allowedHosts;  // Safe to share - can't be modified
            }
        }

        //usage
        ImmutableConfig config = new ImmutableConfig(List.of("api.example.com"));

        // Thread 1: Gets list and reads it
        List<String> hosts1 = config.getAllowedHosts();
        for (String host : hosts1) {
            System.out.println(host);  // Safe!
        }

        // Thread 2: Gets list (same immutable list)
        List<String> hosts2 = config.getAllowedHosts();

- About static factories
    - "..An immutable class can provide static factories (Item 1) that cache frequently requested instances to avoid creating new instances when existing ones would do.."
    - "...static factories causes clients to share instances instead of creating new ones, reducing memory footprint and garbage collection costs. Opting for static factories in place of publicconstructors when designing a new class gives you the flexibility to add caching later, without modifying clients...."

- Sharing freely
    - Immutable objects = can be shared freely = never have to make defensive copies/ any copy
    - Since the copies would be forever equivalent to the originals
    - No need to provide clone or copy constructor inside immutable class

- Cons of immutability
    - they require a separate object for each distinct value. Creating these objects can be costly, especially if they are large. 
    - performance problem, if perform a multistep operation that generates a new object at every step, eventually discarding all objects except the final result.
    * soln
        - Primitive
            - Package-Private Mutable Companion = Internal optimization for predictable operations
            - guess which multistep operations will be commonly required and to provide them as primitives. If a multistep operation is provided as a primitive, the immutable class does not have to create a separate object at each step.
            - example: BigInteger class uses MutableBigInteger internally. Not available for us to use.
        - provide a public mutable companion class.
            - Give users flexibility for unpredictable operations
            - Example: String class = immutable, provides a mutable companion class = StringBuilder.

- Design Alternatives
    - want immutability = gurantee no subclass allowed = make class final
    - alternative: make constructors private/package private, have public static factories instead
