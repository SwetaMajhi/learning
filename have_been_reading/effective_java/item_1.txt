Consider static factory methods instead of constructors

- When thinking of providing ways to instantiate a class:
    - we know public constructors 
    - another way = a static method that returns an instance of the class
    - Example: Boolean class (the boxed primitive class for boolean)
    - A class can provide its clients with static factory methods instead of, or in addition to, public constructors.
    
- 1st Adv of static factory methods
    - Unlike constructors, they have names.
        - code easier to read
        - In cases where a class seems to require multiple constructors with the same signature, replace the constructors with static factory methods and carefully chosen names to highlight their differences.

- 2nd Adv of static factory methods
    - They are not required to create a new object each time they’re invoked
        - This allows immutable classes to use preconstructed instances, or to cache instances as they’re constructed, and dispense them repeatedly to avoid creating unnecessary duplicate objects = performace improve
        - Example:
            public final class Boolean {
                // Pre-constructed instances (immutable and safe to share)
                public static final Boolean TRUE = new Boolean(true);
                public static final Boolean FALSE = new Boolean(false);
                
                private final boolean value;
                
                // Private constructor
                private Boolean(boolean value) {
                    this.value = value;
                }
                
                // Static factory - returns cached instances
                public static Boolean valueOf(boolean b) {
                    return b ? TRUE : FALSE;  // Reuses same objects!
                }
            }

            // Usage
            Boolean b1 = Boolean.valueOf(true);
            Boolean b2 = Boolean.valueOf(true);
            Boolean b3 = Boolean.valueOf(true);

            System.out.println(b1 == b2);  // true (same object!)
            System.out.println(b2 == b3);  // true (same object!)

            // Only TWO Boolean objects exist in entire JVM
            // Safe because Boolean is immutable

- Instance controlled classes
    - when returning same object from repeated invocations = classes maintaining strict control over instances
    - such classes used for 
        - singleton classe
        ** noninstantiable class
        - making sure no 2 equal instances exist.
            - a.equals(b) if and only if a == b.
            - If two references are equal by value (a.equals(b) is true), then they are the same object (a == b is true).
            - This is the basis of the Flyweight pattern.
            - Example:
            //Integer only controls instances in the range -128 to 127. Outside this range, it creates new objects, so the guarantee doesn't hold.
            Integer a = Integer.valueOf(100);
            Integer b = Integer.valueOf(100);

            // Value equality
            System.out.println(a.equals(b));  // true (same value)

            // Reference equality - ALSO TRUE!
            System.out.println(a == b);  // true (same object!)

            Integer x = Integer.valueOf(1000);
            Integer y = Integer.valueOf(1000);

            // Value equality
            System.out.println(x.equals(y));  // true (same value)

            // Reference equality - FALSE
            System.out.println(x == y);  // false (different objects)
    
    - we see similar behaviour in Enums also. JVM ensures only ONE instance of each enum constant exists.
        public enum Color {
            RED, GREEN, BLUE
        }

        // Usage
        Color c1 = Color.RED;
        Color c2 = Color.RED;

        // Value equality
        System.out.println(c1.equals(c2));  // true

        // Reference equality - ALSO TRUE!
        System.out.println(c1 == c2);  // true

- 3rd Adv of static factory methods
    - Static factory methods can return an object of any subtype
                                // With static factory, you can return ANY subtype
                                public abstract class Animal {
                                    // Returns Animal, but could be Dog, Cat, Bird, etc.
                                    public static Animal create(String type) {
                                        switch (type) {
                                            case "dog": return new Dog();      // Dog extends Animal
                                            case "cat": return new Cat();      // Cat extends Animal
                                            case "bird": return new Bird();    // Bird extends Animal
                                            default: throw new IllegalArgumentException();
                                        }
                                    }
                                }

                                class Dog extends Animal { /* ... */ }
                                class Cat extends Animal { /* ... */ }
                                class Bird extends Animal { /* ... */ }

                                // Usage
                                Animal pet = Animal.create("dog");  // Returns Dog, but type is Animal
                                // Caller doesn't need to know about Dog class!

    - But what is up with "noninstantiable companion class"!!
        - Like the "Collections" class above is public but has private constructor and none of its public static methods are creating/returning any object.
        - But why! Before Java 8, interfaces couldn't have static methods, so a separate companion class was needed to hold static factory methods. Java doesn't have "static classes" or "utility classes" as first-class concepts. A noninstantiable class is the workaround.
        -  After Java 8, interfaces can have static methods, eliminating the need for companion classes, I think!
        - Example: Can put static methods directly in interface!
            public interface List<E> extends Collection<E> {
                
                // Instance methods
                boolean add(E e);
                E get(int index);
                
                // Static factory methods - directly in interface! ✓
                static <E> List<E> of() {
                    return ImmutableCollections.emptyList();
                }
                
                static <E> List<E> of(E e1) {
                    return new ImmutableCollections.List1<>(e1);
                }
                
                static <E> List<E> of(E e1, E e2) {
                    return new ImmutableCollections.List2<>(e1, e2);
                }
            }

            // Usage - much more intuitive!
            List<String> empty = List.of();  // Directly from interface!
            List<String> one = List.of("hello");
            List<String> two = List.of("hello", "world");

            // No need for separate Lists companion class!
        - Then Why have an interface at all? Why not just have a class with static methods?
